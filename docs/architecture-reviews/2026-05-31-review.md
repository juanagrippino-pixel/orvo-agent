## 1. OperationalCase / WorkItem alignment with Atlassian patterns

### `codex/work-management` — merge-ready, with Jira-parity gaps

What aligns:

- Adds richer workflow states: `open`, `acknowledged`, `in_progress`, `resolved`, `dismissed` in `app/brain/operational_cases.py:22`.
- Defines deterministic allowed transitions in `_CASE_STATUS_TRANSITIONS` at `app/brain/operational_cases.py:43-46`.
- Treats resolved/dismissed recurrences as deterministic reopens with preserved timeline/evidence, at `app/brain/operational_cases.py:489-527`.
- Adds operator actions in `app/brain/operator_api.py:23` and maps them to lifecycle transitions in `app/brain/operator_api.py:2755-2763`.
- Extends case views/JQL to include `in_progress_cases` and in-progress queue views, matching Atlassian-style saved filters and issue queues.
- Keeps OperationalCase as the source of truth; reports/WhatsApp/operator views remain projections.

Gaps / follow-ups:

- **No project abstraction yet.** Jira-like project keys, project-scoped issue numbering, and tenant/project configuration are still absent. `business_id` is still the effective project boundary.
- **No explicit issue type entity.** `case_type` behaves like an issue type, but there is no first-class issue type registry/versioning layer mapped to `docs/specs/d2c-case-family-catalog.md`.
- **No status category layer.** Statuses exist, but there is no separate Atlassian-style category (`To Do`, `In Progress`, `Done`) for rollups and workflow-independent analytics.
- **Manual resolve reason is still optional.** The contract says manual resolve/dismiss requires a reason, but `apply_case_action` only requires a non-empty reason for `dismiss_case` at `app/brain/operator_api.py:2762-2763`; `resolve_case` falls back to a default reason.
- **Open → resolved was removed from the documented transition table.** This may be acceptable for stricter workflow control, but it is a contract change and should be explicit in release notes/ADR if merged.

Recommendation: **Merge-ready** because it improves workflow determinism, timeline auditability, JQL-style operator queues, and source-of-truth discipline without bypassing platform contracts. Track the Jira-parity gaps above as follow-up architecture tasks.

## 2. Semantic registry canonical source of truth / metric drift

### Cross-branch assessment

- No branch introduces a competing metric catalog as the source of truth.
- `codex/work-management` continues to import/use `CASE_FAMILY_METRICS`, `default_metric_registry`, and `validate_metrics` from `app.brain.semantics` in `app/brain/operational_cases.py:20`.
- `codex/connector-platform` keeps emitted metric metadata at the connector-family level while metric-key validation stays in the semantic metric registry. This is aligned with the connector registry contract: connector specs advertise capabilities/families; metrics remain canonical in the metric registry.

Gaps / follow-ups:

- Connector specs still list emitted metric families as strings, e.g. `commerce.orders`, `commerce.revenue`, `runtime.freshness`; they should eventually reference semantic registry family IDs/constants to avoid typo drift.
- Merge gates should include tests that fail on connector-declared emitted families that are not present in the semantic registry.

Recommendation: **No blocker found** on semantic registry canonicality for the reviewed diffs.

## 3. Connector platform separation: adapter / service / storage

### `codex/connector-platform` — merge-ready, incremental

What aligns:

- Moves daily-report invocation toward registry metadata instead of ad hoc connector branching.
- Adds allowlisted executor metadata in `app/brain/connector_registry.py:117-141` with static `adapter_module`, `report_factory`, supported runtime modes, and declarative factory params.
- Builds adapter kwargs through declarative `ConnectorFactoryParam` bindings at `app/brain/connector_registry.py:101-114` and `app/brain/connector_registry.py:226-308`.
- The dynamic import path is not tenant-controlled; it is sourced from static registry specs in `DEFAULT_CONNECTOR_SPECS` (`app/brain/connector_registry.py:608-839`). This is an acceptable allowlist pattern.
- Pipeline now has `_build_daily_report_for_connector_type` at `app/brain/pipeline.py:300-333`, which validates daily-report capability, finds the enabled connector, builds kwargs from the registry, calls the factory, and enforces `DailyReport` return type.
- Multi-connector pipeline path dispatches once after merging reports, preserving dispatcher/idempotency discipline (`app/brain/pipeline.py:336-380`).

Gaps / follow-ups:

- Legacy connector-specific pipeline functions still exist and are hardcoded. That is acceptable during migration, but the compiled runtime should eventually use the registry path as the default.
- OAuth/API connector executors still bind legacy inline `access_token` out of connector params for Tiendanube, MercadoLibre, and Meta Ads (`app/brain/connector_registry.py:701-703`, `755`, `817-819`). The registry documents `required_secret_refs`, but runtime secret-ref resolution is not implemented here.
- `service_bindings` are passed as a fixed set of service keys from pipeline to registry (`app/brain/pipeline.py:323-328`). This is safe, but compiled runtime should formalize these bindings rather than expanding ad hoc kwargs over time.

Recommendation: **Merge-ready** as a connector-platform migration step. Do not market it as full connector runtime compilation until secret-ref resolution and registry-first execution are complete.

## 4. Workflow automation: idempotency, approval gates, audit

### Work-management

- Lifecycle transitions append timeline events (`app/brain/operational_cases.py:544-568`), preserving audit chronology.
- Deterministic recurrences reopen resolved/dismissed cases rather than silently changing closed cases (`app/brain/operational_cases.py:489-527`).
- Operator actions are explicit action keys, not free-form lifecycle mutations (`app/brain/operator_api.py:23`, `2755-2763`).

Follow-ups:

- Add explicit approval-gate metadata for destructive/external actions as action-key catalog work evolves.
- Enforce non-empty reason for `resolve_case`, not only `dismiss_case`.

### Trust/Admin/Security

What aligns:

- Adds an operator audit table in `app/brain/storage.py` and an append helper in `app/brain/operator_audit.py:20-65`.
- Audit writes redact event payloads before persistence (`app/brain/operator_audit.py:42`) and redact actor/business/target text fields.
- Successful internal case actions write audit events from `server.py:345-348` and include action/result metadata.

Blockers / gaps:

- **Audit is success-only.** In `server.py:333-344`, `apply_case_action` runs before `audit_store.append_event`; if the action key is unknown, scope check fails, transition is invalid, or payload validation fails, no audit event is written. Security-relevant denied/failed attempts should be audited.
- **Auth failures are not audited.** `_authorize_internal_operator` at `server.py:134-145` returns errors before any audit store is available.
- **No RBAC/action-scope enforcement.** `_authorize_internal_operator` only validates one bearer token from `ORVO_INTERNAL_OPERATOR_TOKEN`; `X-Orvo-Operator` is treated as actor identity but not authenticated/authorized by role, project, action, or business scope.
- **Append-only is by convention.** There are no SQLite triggers/guards to prevent update/delete of `operator_audit_events`; no code path mutates it, but the storage contract is not technically enforced.

Recommendation: **Needs work** before merging as a Trust/Admin/Security branch. The audit store is useful, but the branch should at minimum audit failed/denied attempts and define a minimal RBAC/action-scope boundary or explicitly document that RBAC is out of scope for the branch.

## 5. Trust/Admin/Security and secret boundaries

Positive findings:

- No raw credentials were found in the reviewed diff content.
- Connector registry secret requirements are modeled as secret refs/handles, while legacy inline secret fields are explicitly labeled transitional.
- Operator audit payloads and API projections redact secret-like values before persistence/response.
- Internal endpoints remain bearer-token protected and business-scoped in query/action helpers.

Strategic gaps:

- Runtime still has transitional inline secret paths for external connector adapters.
- Internal operator auth is not true RBAC; it is a shared-token gate.
- Audit coverage is incomplete for rejected attempts, auth failures, and potentially reads of sensitive internal projections.
