## 1. OperationalCase / WorkItem alignment with Atlassian patterns

### Current alignment

The canonical `OperationalCase` model is directionally aligned with an Atlassian/Jira-style work item:

- It explicitly states Operational Cases are deterministic, evidence-backed work items and the control-plane source of truth; WhatsApp/report surfaces are projections (`app/brain/operational_cases.py:1-6`).
- It has issue-type-like `case_type` values (`sales_drop`, `stockout_risk`, `spend_without_orders`, `data_stale`, `unanswered_conversations`, `channel_mix_shift`) (`app/brain/operational_cases.py:26-33`).
- It has workflow statuses and terminal states: `open`, `acknowledged`, `in_progress`, `resolved`, `dismissed` (`app/brain/operational_cases.py:22-25`).
- It has deterministic status transitions (`open → acknowledged|in_progress|dismissed`; `acknowledged → in_progress|resolved|dismissed`; `in_progress → resolved|dismissed`; terminal closed statuses have no outgoing transition) (`app/brain/operational_cases.py:47-53`).
- It has a timeline model with events such as `case_opened`, `case_updated`, `case_reopened`, `status_changed`, `case_assigned`, `evidence_attached`, `operator_comment` (`app/brain/operational_cases.py:34-45`).
- It has a safe JQL-lite query layer that intentionally does not translate user input into SQL (`app/brain/operator_views.py:1-5`). The parser allowlists fields/operators, rejects SQL-like constructs, caps query length/clauses, keeps business scoping outside the user query, and projects built-in readonly views (`app/brain/operator_views.py:71-84`, `151-198`).

### Branch: `codex/work-management` — merge-ready incremental

What improved:

- Requires non-empty reasons for terminal operator actions. `resolve_case` and `dismiss_case` now fail with `missing_case_action_reason` when no reason is provided (`app/brain/operator_api.py:2929-2936` on branch `codex/work-management`). This closes the previous auditability gap for manual terminal transitions.
- Adds focused regression coverage for terminal transition guards (`f8fd2f8`, `63d186c`).
- Preserves the invariant that operator APIs call the canonical case store rather than owning lifecycle state.

Remaining Atlassian/Jira parity gaps:

- **No Project entity yet.** `business_id` is the effective project boundary, but there is no project key, project-scoped issue numbering, project component/version model, or project workflow scheme.
- **No first-class IssueType registry.** `case_type` is issue-type-like, but it is not yet a registry/versioned catalog tied directly to `docs/specs/d2c-case-family-catalog.md`.
- **No Status Category abstraction.** The system has statuses, but not Jira-style categories (`To Do`, `In Progress`, `Done`) that decouple rollups from workflow details.
- **JQL-lite is correct for safety but intentionally narrow.** It supports operational filters, not saved custom filters, user favorites, boards, SLA queries, or cross-project search.

ARB disposition: **merge-ready as a work-management increment**. Track Project, IssueType registry, StatusCategory, and saved filter/board abstractions as platform follow-ups.

## 2. Semantic registry as canonical source of truth / metric drift

### Current alignment

- Operational Cases import semantic registry primitives from `app.brain.semantics`: `CASE_FAMILY_METRICS`, `default_metric_registry`, and `validate_metrics` (`app/brain/operational_cases.py:20`).
- Case evidence construction validates report metrics against the registry and case-family allowlists (`app/brain/operational_cases.py:840-842`).
- The metric registry defines `CASE_FAMILY_METRICS` as the D2C case-family contract (`app/brain/semantics/metric_registry.py:159-160`).
- Contract tests guard that case-family mappings reference registered case-allowed metrics (`tests/contracts/test_metric_registry_contract.py:130-144`).
- Alias behavior is explicit: legacy `avg_order_value` maps to `commerce.average_order_value`, and tests verify non-case-allowed aliases cannot silently drive case lifecycle (`tests/contracts/test_metric_validation_contract.py:780-816`).

### Drift risks found

- Connector specs still advertise emitted metric families as raw strings (for example `commerce.revenue`, `commerce.average_order_value`, `runtime.freshness` in `app/brain/connector_registry.py:681-683`). This is acceptable at current scale, but the next platform step should validate connector-declared families against semantic registry families/constants.
- Branches adding projections (`search-analytics`, `operator-surfaces`, `workflow-automation`) do not introduce competing metric sources of truth; they derive from case/evidence snapshots.

ARB disposition: **no merge blocker on semantic canonicality** for reviewed branches. Add a connector-family-to-semantic-registry contract test before the connector platform expands.

## 3. Connector platform separation: adapter / service / storage

### Branch: `codex/connector-platform` — merge-ready incremental

What aligns:

- `compile_business_runtime` remains pure and side-effect free, using the connector registry as source of truth (`app/brain/runtime.py:145-156`).
- Compiled connector runtime artifacts include public params, deterministic secret refs, capabilities, emitted metric families, supported runtime modes, executor factory path, health policy, scopes, rate-limit policy, and lifecycle metadata (`app/brain/runtime.py:39-59`, `252-274`).
- Raw legacy secret fields are stripped from the compiled artifact; secret refs are emitted as `secret://businesses/{business_id}/connectors/{connector_id}/{secret.name}` (`app/brain/runtime.py:356-369`).
- Runtime mode enforcement is registry-driven: unsupported `run_mode` fails compilation (`app/brain/runtime.py:233-240`).
- Health/rate/lifecycle metadata is copied from registry specs into compiled runtime (`app/brain/runtime.py:279-300`).

Remaining gaps:

- **No real secret-ref resolver/execution boundary yet.** The compiled artifact is safe, but current adapters can still receive legacy resolved secrets outside this compiler. This is an accepted transitional state, not a complete secret-boundary implementation.
- **Adapter/service/storage separation is improving but not complete.** The compiler knows registry metadata; execution still needs a formal service binding layer so runtime executors consume compiled connector contracts rather than ad hoc kwargs.
- **Connector-emitted families need semantic registry validation.** See metric drift section.

ARB disposition: **merge-ready as a compiled-runtime/connector metadata increment**. Position as groundwork, not full connector platform completion.

## 4. Workflow automation: idempotency, approval gates, audit

### Branch: `codex/workflow-automation` — merge-ready as dry-run only

What aligns:

- The module explicitly defines itself as service-layer dry-run/projection code and does not mutate case state, dispatch messages, or call external systems (`app/brain/workflow_automation.py:1-7`).
- Rules are scoped to the case business; business mismatch fails closed (`app/brain/workflow_automation.py:184-186`).
- Action keys are allowlisted in `WORKFLOW_ACTION_REGISTRY`; unknown workflow actions fail before condition evaluation (`app/brain/workflow_automation.py:41-60`, `184-195`, `303-309`).
- Planned actions include deterministic idempotency keys derived from business/rule/case/action/redacted params (`app/brain/workflow_automation.py:121-139`, `205-240`).
- Duplicate planned actions are suppressed by idempotency key (`app/brain/workflow_automation.py:275-300`).
- External/destructive actions can be marked `approval_required` and return `blocked_approval_required`; suggestions remain non-mutating (`app/brain/workflow_automation.py:197-202`).
- `side_effects_executed` is always `0` in the projection output (`app/brain/workflow_automation.py:321-338`).

Gaps before real execution:

- No persisted workflow rule registry, rule versioning, approval object, approval transition, or audit-store integration exists yet.
- Idempotency is computed, but not reserved in durable storage because the branch intentionally does not execute side effects.
- There is duplicated action-key metadata across workflow automation and operator action catalog work; this should converge into a single action-key catalog/service.

ARB disposition: **merge-ready only if labeled as dry-run simulation**. Do not wire it to execute mutations/external actions until approval, durable idempotency, RBAC, and audit contracts are implemented.

## 5. Trust/Admin/Security: RBAC, audit trail, secret boundaries

### Branch: `codex/trust-admin-security` — needs work before security-complete merge

What aligns:

- Adds explicit internal role helpers: `viewer`, `operator`, `admin`; `viewer` lacks `case:action`; unknown roles fail closed (`app/brain/operator_auth.py:15-23`, `47-85`).
- Adds an append helper for redacted operator audit events (`app/brain/operator_audit.py:20-65`).
- Adds a durable `operator_audit_events` table and indexes (`app/brain/storage.py:108-124`).
- Audits internal auth denials before returning 401/503 (`server.py:140-193`).
- Audits authorization denials for case actions (`server.py:381-406`).
- Audits successful case actions with redacted reason/comment/metadata/result payloads (`server.py:408-438`; audit helper redacts at write time in `app/brain/operator_audit.py:42-61`).

Gaps / blockers for Trust/Admin/Security claim:

- **Post-authorization action failures are not audited.** If `apply_case_action` rejects an unknown action, cross-tenant case, invalid transition, missing reason, or invalid payload after RBAC passes, `_with_internal_stores` converts the error to an API response but no audit event is written (`server.py:408-418` followed by success-only audit at `418-438`). Denied/failed mutation attempts are security-relevant.
- **Default role is `operator`.** Missing role defaults to mutation-capable `operator` for compatibility (`app/brain/operator_auth.py:24`, `47-56`). This is convenient but over-privileged for a Trust/Admin boundary; new deployments should default to `viewer` or require explicit role claims.
- **RBAC is role-only, not tenant/project scoped.** There is no principal-to-business/project entitlement model. A bearer-token holder with `operator` role can target any `business_id` accepted by the route.
- **Audit immutability is by convention.** The helper only inserts, but SQLite schema does not prevent updates/deletes through other code paths; no triggers or storage API constraints enforce append-only semantics.
- **Audit read/admin surface is absent.** There is no reviewed query/export endpoint for audit trails, retention policy, or admin RBAC separation.
- **Shared bearer token remains the authentication root.** This is acceptable for internal alpha but not sufficient for a sellable admin/security story.

ARB disposition: **needs work**. Consider cherry-picking the audit schema/helper and role scaffolding only if paired with follow-up fixes for failed-attempt audit, default least privilege, scoped principals, and append-only enforcement.

### Secret boundary summary

Positive:

- Operational Case fields and metadata redact text/URIs/secrets (`app/brain/operational_cases.py:80-82`, `270-295`).
- Runtime compilation strips raw connector secrets and emits secret refs (`app/brain/runtime.py:252-369`).
- Operator projections and audit data use `redact_secrets` before response/persistence.

Remaining strategic gap:

- The runtime still has transitional legacy inline secret paths at adapter execution time. This must be closed before external connector management is exposed as a tenant/admin feature.

## 6. Operator surfaces and search/analytics

### Branch: `codex/search-analytics` — merge-ready incremental

What aligns:

- Adds `source_connector` and `degraded` to allowlisted JQL-lite fields without allowing arbitrary SQL or unscoped business queries (`app/brain/operator_views.py:72-87` on branch).
- Adds a built-in `connector_degraded` view over actionable cases whose evidence is stale/degraded/missing (`app/brain/operator_views.py:140-146`).
- Uses projection helpers for source connectors and degraded detection (`app/brain/operator_views.py:22`, `313-362`).
- Keeps query evaluation in memory over `store.list_cases(business_id=...)`, preserving business scope and store authority (`app/brain/operator_views.py:185-198`).

Risks / follow-ups:

- In-memory filtering is fine for first tenants; larger tenants will need an indexed query service that preserves the same JQL allowlist and business-scope invariants.
- `source_connector` is derived from evidence snapshots; this is correct, but connector source labels should be normalized through the connector registry over time.

ARB disposition: **merge-ready incremental**.

### Branch: `codex/operator-surfaces` — needs rebase/changes

What aligns:

- Adds an internal case action catalog endpoint/projection (`app/brain/operator_api.py:2960-3056` on branch) and docs/server/test updates.
- The catalog distinguishes API-enabled manual mutations from disabled/approval-required external actions (`request_external_action` is disabled and `approval_required=True` at `app/brain/operator_api.py:3042-3049`).
- Transport remains thin and calls operator API helpers rather than owning case state.

Issues to fix before merge:

- **Catalog drift:** the branch says `resolve_case` has `requires_reason: False` (`app/brain/operator_api.py:3022-3028`), but the newer work-management branch correctly requires non-empty reasons for all terminal statuses. This must be rebased/aligned.
- **Action metadata duplication:** case action keys now appear in operator API, workflow automation registry, and docs. A single action-key catalog/service should generate operator catalog projections and workflow allowlists.
- **Approval is descriptive only.** Disabled approval-required actions are safe, but there is no approval object/workflow/audit transition yet.

ARB disposition: **needs work / rebase** before merge unless cherry-picking only non-conflicting docs/tests.

## 7. Strategic recommendations / integration order

1. **Merge first:** `codex/work-management`, `codex/connector-platform`, `codex/search-analytics` after normal rebase and full-suite CI. These are incremental and preserve source-of-truth boundaries.
2. **Merge as dry-run service only:** `codex/workflow-automation`, with explicit release notes that it executes zero side effects and is not an automation executor.
3. **Rework before merge:** `codex/trust-admin-security` should audit failed action attempts, move toward least-privilege default role, add business/project-scoped principal checks, and enforce append-only audit semantics.
4. **Rebase before merge:** `codex/operator-surfaces` must align terminal reason requirements and avoid action catalog duplication.
5. **Next platform contracts to add:**
   - Project / IssueType / StatusCategory registry contract.
   - Single D2C action-key catalog service used by operator surfaces and workflow automation.
   - Connector emitted-family validation against the semantic metric registry.
   - Durable workflow approval and idempotency ledger.
   - Audit trail query/export contract with RBAC and retention.
