## 1. OperationalCase / WorkItem against Atlassian patterns

### What is aligned

Current `OperationalCase` is an explicit work item and not just report text:

- Durable identity and tenant scope: `case_id`, `business_id`, `dedupe_key` with SQLite uniqueness on `(business_id, dedupe_key)`.
- Issue-type analogue: `case_type` values such as `sales_drop`, `stockout_risk`, `data_stale`, `fulfillment_backlog`, `unanswered_conversations`.
- Workflow/status lifecycle: `open → acknowledged → in_progress → resolved/dismissed`, with terminal states enforced and timestamps validated.
- Audit-like timeline: `case_opened`, `case_updated`, `case_reopened`, `status_changed`, `case_assigned`, `evidence_attached`, `operator_comment`.
- Evidence-backed fields: snapshots, evidence refs, artifact refs, run ids, source run ids, semantic evidence metrics.
- Operator query layer: `operator_views.py` provides safe JQL-lite parsing over in-memory case projections; it does not compile user input to SQL.
- Built-in read-only views exist for open, acknowledged, in-progress, resolved, critical open, data stale, stock risks, and degraded connector cases.
- Action catalog is central for API-enabled/manual/suggestion/approval-required action keys.

This is close to Jira’s core pattern: issue/work item + issue type + workflow transitions + query/projection layer. It is also correctly deterministic: cases are created/updated from evidence and runtime outputs, not from LLM copy.

### Architectural gaps

1. **No explicit Project model yet.** `business_id` is currently doing tenant/project duty. That is acceptable for the narrow D2C wedge, but an Atlassian-like control plane should introduce a first-class project/workspace model before multi-team/operator expansion. Recommended fields: `project_key`, `project_id`, `business_id`, `display_name`, `case_type_scheme_id`, `workflow_scheme_id`.

2. **No status categories.** Jira distinguishes concrete statuses from status categories (`To Do`, `In Progress`, `Done`). Current statuses are reasonable, but there is no canonical `status_category`. Dashboards therefore derive “actionable” from status sets. Add a small immutable mapping: `open -> todo`, `acknowledged/in_progress -> in_progress`, `resolved/dismissed -> done`.

3. **No workflow scheme registry.** `_CASE_STATUS_TRANSITIONS` is hardcoded globally. Good for v1, but a platform core should eventually expose a registry/scheme per case type. Do not make this tenant-customizable yet; first add a deterministic internal `WorkflowDefinition` registry.

4. **JQL-lite lacks project/status-category fields.** The JQL-lite implementation is intentionally safe and useful, but not yet Jira-like enough for operator scale. Add fields only after canonical models exist: `project`, `status_category`, `assignee`, perhaps `case_family`/`issue_type` aliasing.

5. **Case type vs issue type naming is mixed.** Product docs can still use “Operational Case,” but platform internals should decide whether `case_type` is the stable issue type key or whether a formal `IssueType`/`WorkItemType` registry wraps it. Avoid parallel names that drift.

### Recommendation

Keep `OperationalCase` as the source-of-truth work item for now. Add project/status-category/workflow-scheme primitives as additive registry layers, not as broad rewrites. Avoid adding custom workflow execution until status categories and action approval states are modeled cleanly.

## 2. Semantic registry and metric drift

### What is aligned

The current semantic registry is the right canonical source:

- `MetricRegistry` owns canonical definitions, aliases, allowed sources, family, unit, aggregation, freshness requirements, report/case permissions, and PII class.
- `CASE_FAMILY_METRICS` ties detectable case families to semantic metrics.
- Aliases explicitly map legacy adapter metric keys like `orders_today`, `revenue_today`, `stock_units`, etc., to canonical keys like `commerce.orders.count`, `commerce.revenue.total`, and `commerce.inventory.available_units`.
- Connector registry imports metric validation helpers and validates emitted metric families/sources.
- Focused metric/adapter contracts passed in the current baseline.

### Drift risks observed

1. **Adapters still emit legacy metric keys.** This is intentionally bridged by aliases, but it remains a drift vector. The registry is canonical, but adapter code still visibly constructs metrics with keys such as `revenue_today`, `orders_today`, `stock_units`, `ad_spend_today`. Keep this only as transitional; new adapters should emit canonical keys directly or be required to declare alias use in tests.

2. **Operational case dedupe suffixes include string metric families.** `operational_cases.py` has fallback suffixes like `commerce.revenue/daily`, `commerce.inventory/daily`, `ads.spend/daily`. They are not creating new metrics, but they can drift from the semantic registry vocabulary. Prefer helper functions that derive suffixes from `CASE_FAMILY_METRICS` or canonical family metadata.

3. **Branch risk: stale connector/work-management branches mutate `metric_registry.py`.** Several stale branches show large diffs against current baseline and can accidentally remove WooCommerce allowed sources or recent metric contracts. Any branch touching `metric_registry.py` should run the three metric contract tests plus adapter emission contracts before merge.

### Recommendation

Promote a “no raw metric key literals outside registry/tests/adapters” rule. For adapter-local legacy aliases, require a contract test proving every emitted key resolves through `default_metric_registry()`. For Operational Cases, build evidence/dedupe metric references through registry helpers.

## 3. Connector platform separation: adapter / service / storage

### What is aligned

The current baseline has the right separation direction:

- `connector_registry.py` defines `ConnectorSpec`, secret requirements, executor metadata, capabilities, scope/rate-limit/lifecycle metadata, and validation helpers.
- Adapter modules remain under `app/brain/adapters/*` and expose daily-report factories.
- Runtime/runner code compiles business runtime and routes connector execution through control-plane metadata rather than one-off endpoints.
- Storage concerns live in `storage.py`, `run_ledger.py`, `operational_cases.py`, and audit stores rather than in adapter modules.
- Secret boundaries exist through required secret refs and redaction tests; current baseline includes WooCommerce and secret-ref hardening.

### Gaps and risks

1. **`ConnectorExecutorMetadata.load_factory()` should retain factory contract validation.** Current baseline uses worker contracts; stale `codex/connector-platform` removes `validate_daily_report_adapter` and only checks `callable`. That is a regression. Keep structural validation for adapter factories.

2. **Service bindings are still passed as loose kwargs.** This is acceptable for current adapters but should eventually become a typed connector execution context: HTTP clients, secret resolver, clock, retry/rate-limit policy, and run ledger handle.

3. **Connector branches must preserve secret-ref flow.** `codex/connector-platform` diff removes current secret resolver/worker-contract files and WooCommerce in the tree comparison. Do not merge that branch as-is.

4. **Registry is doing both catalog and validation.** That is okay for v1, but as connectors grow, split pure registry definitions from execution service if the file becomes too large. Do not bypass registry to add endpoint-specific connector shortcuts.

### Recommendation

For connector work, rebase on baseline and make changes through `ConnectorSpec` + runtime execution plan + run ledger. Preserve adapter factory validation, secret resolver contracts, and current WooCommerce support.

## 4. Workflow automation: idempotency, approvals, audit

### Current baseline

The baseline already has strong foundations:

- `WorkflowActionLedgerStore` records planned actions with durable idempotency keys.
- Approval-required actions are blocked and create approval requests rather than executing side effects.
- Simulation returns `side_effects_executed = 0`.
- Action keys come from the central action catalog.
- Params are redacted before persistence/projection.
- Duplicates are skipped by idempotency key.

### `codex/workflow-automation` review

This branch is architecturally aligned. It adds:

- Approval decision model (`approved` / `rejected`) without executing the action.
- `WorkflowApprovalDecisionRecord` with `side_effects_executed = 0`.
- Cross-business approval lookup protection.
- Double-decision prevention.
- Redacted actor/ref/reason in decision audit event.
- Required action-param validation based on action catalog (`reason`, `comment`, `assignee_ref`, etc.).
- Explicit approval gate projection for approval-required actions.

ARB verdict: **merge-ready after normal test run on the branch**. `git merge-tree origin/feat/orvo-brain-control-plane codex/workflow-automation` showed no conflicts. Its behavior strengthens the deterministic control plane and keeps execution behind approval/idempotency gates.

### Remaining workflow gap

Approval decisions currently move approved actions to `pending_execution`; there is still no governed executor. That is correct. Next executor work must add:

- explicit executor identity and permissions;
- durable execution attempt ledger;
- idempotent external side-effect keys;
- approval-to-execution audit linkage;
- dry-run/simulation parity;
- failure/retry semantics;
- no direct WhatsApp/report state mutation.

## 5. Trust / Admin / Security

### What is aligned

Current baseline has a useful first RBAC/audit slice:

- Internal operator auth separates token authentication from role/permission checks.
- Roles: `viewer`, `operator`, `admin`.
- Permissions: currently `internal:read` and `case:action` in baseline; audit-export branch adds `operator_audit:read`.
- `operator_session` projection redacts actor refs and exposes capabilities.
- Case actions enforce mutation permission and write audit events for failures/denials.
- `SQLiteOperatorAuditStore` persists redacted audit events and scopes list queries by `business_id`.
- Internal API errors use safe envelopes with `redaction_applied=true`.

### `codex/eng-factory-audit-export-admin-20260602` review

This branch is strategically aligned. It adds an admin-only endpoint:

`GET /internal/brain/businesses/{business_id}/operator-audit-events`

and introduces `operator_audit:read` only for `admin`, with tests for operator denial and redacted admin export. `git merge-tree` showed no conflicts.

ARB verdict: **merge-ready after rebase/test, with minor polish**. The branch is 6 commits behind the remote baseline and should be rebased; also fix the missing trailing newline in the new route file.

### Gaps

1. **Tenant scoping is header/token-level, not identity-provider-level.** The route scopes by `business_id`, but there is not yet a durable operator-to-business grant model. Before external/live exposure, add a scoped authorization store.

2. **Audit store redacts before persistence.** This is safe for secrets but can reduce forensic fidelity. For this product stage, safety wins. If stronger forensic audit is needed later, use encrypted privileged raw audit payloads with strict break-glass flow—not plaintext.

3. **Auth denials should be audited consistently.** Some trust branches attempt this. Rebase and replay onto current `app/http/internal_brain/common.py` and `operator_auth.py` rather than reviving the old monolithic `operator_api.py`.

4. **RBAC permission names should become registry-backed.** Keep constants now, but when permissions grow, introduce a permission registry and tests verifying role projections.

## Strategic architecture recommendations

1. **Add a Work Management registry layer.** Minimal additive models: `Project`, `IssueType`/`CaseTypeDefinition`, `WorkflowDefinition`, `StatusDefinition`, `StatusCategory`. Keep D2C defaults hardcoded/registered; do not introduce tenant-custom workflow customization yet.

2. **Turn JQL-lite into a stable operator contract.** Keep the safe parser. Add `project`, `status_category`, `assignee_ref`, and `case_type`/`issue_type` aliases only after registry models exist. Persist saved filters later; built-ins are enough now.

3. **Keep semantic registry as the only metric authority.** New detections and adapters must validate against `default_metric_registry()`. Avoid metric strings in case logic unless derived from registry constants.

4. **Protect connector contracts.** New connector branches must preserve adapter/service/storage separation: adapter factory, connector spec, runtime plan, run ledger, secret resolver, redaction. No endpoint/runtime shortcut should directly call Tiendanube/WooCommerce/WhatsApp outside compiled runtime contracts.

5. **Finish approval before execution.** `codex/workflow-automation` moves the platform toward approval decisions. The next slice should be an execution-attempt ledger, not direct external effects.

6. **Rebase stale branches before review.** Branches with old `app/brain/operator_api.py` should be considered structurally obsolete. Replaying intent onto the current package layout will reduce merge risk and prevent deletion of current control-plane files.

## Final ARB position

Current baseline is directionally sound and increasingly platform-like. The best near-term integration candidates are `codex/workflow-automation`, `codex/eng-factory-audit-export-admin-20260602`, and the two docs branches. The stale connector/work-management/trust/operator-surface branches should not be merged wholesale; salvage the invariant or endpoint they add, then replay on the current control-plane core.
