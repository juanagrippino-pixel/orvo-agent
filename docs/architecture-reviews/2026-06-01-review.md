## 1. OperationalCase / WorkItem alignment with Atlassian patterns

### Current alignment

The current base is now substantially closer to a Jira-like work-management core:

- `OperationalCase` is explicitly the control-plane source of truth; WhatsApp/report text is only a projection (`app/brain/operational_cases.py:1-6`).
- Work-item lifecycle exists: `open`, `acknowledged`, `in_progress`, `resolved`, `dismissed`, with actionable and terminal status sets (`app/brain/operational_cases.py:23-27`).
- Deterministic transitions prevent arbitrary lifecycle mutation (`app/brain/operational_cases.py:49-55`).
- Timeline events cover opened/updated/reopened/status changes/assignment/evidence/comment events (`app/brain/operational_cases.py:38-47`).
- Operator APIs expose scoped case queues, details, timelines, action application, run history, built-in views, and JQL-lite query delegation (`app/brain/operator_api.py`).
- The action catalog is now centralized in `app/brain/action_catalog.py`, and operator/workflow projections consume it instead of maintaining fully separate action lists.

### Remaining Atlassian/Jira gaps

These are not blockers for a D2C pilot, but they are blockers for claiming Atlassian/Jira parity:

1. **No first-class Project entity.** `business_id` acts as the tenant/project boundary, but there is no project key, project-scoped issue numbering, project settings, components, or workflow scheme.
2. **No IssueType registry.** `OperationalCaseType` is a Python `Literal`, not a versioned registry tied to `docs/specs/d2c-case-family-catalog.md`.
3. **No core StatusCategory.** Some branches introduce `status_category` as projection data, but core status categories (`to_do`, `in_progress`, `done`, perhaps `waiting`) are not first-class workflow metadata.
4. **JQL-lite is safe but narrow.** It is the right first move, but saved filters, boards, custom fields, SLA queries, and indexed search remain future work.

### Branch dispositions

- `codex/service-management` adds useful Jira Service Management-like read-only projections: service record type, owner-facing status, SLA clocks, and escalation reasons. It preserves the invariant that `OperationalCase` remains source of truth. Rebase before merge; promote reusable `status_category` definitions into the core workflow registry rather than leaving them only in this projection.
- `codex/work-management` should be rebased and considered for merge because the owner-facing brief filter keeps internal-only cases out of WhatsApp. But it must not be treated as the canonical fix for case-family eligibility.

## 2. Semantic registry canonicality / metric drift

### Good alignment

- The semantic registry defines canonical metric definitions, aliases, evidence/freshness/report/case flags, PII classes, and case-family mappings (`app/brain/semantics/metric_registry.py`).
- `CASE_FAMILY_METRICS` is the intended source of truth for case-family evidence requirements (`app/brain/semantics/metric_registry.py:159-193`).
- Operational Case evidence snapshots filter report metrics through registry resolution and `CASE_FAMILY_METRICS` before attaching metrics (`app/brain/operational_cases.py:872-901`).
- The registry has rich validators for report, case, surface, connector, freshness, evidence, value-kind, and money-currency envelopes (`app/brain/semantics/metric_registry.py:1020-1380`).

### Critical drift found

There is still a case-family/catalog drift in the current base:

1. **`channel_mix_shift` is detectable but has no `CASE_FAMILY_METRICS` contract.**
   - `OperationalCaseType` includes `channel_mix_shift` (`app/brain/operational_cases.py:28-35`).
   - `_case_type_for_insight` maps titles containing `tiendanube` or `canal` to `channel_mix_shift` (`app/brain/operational_cases.py:823-837`).
   - `CASE_FAMILY_METRICS` does **not** define `channel_mix_shift` (`app/brain/semantics/metric_registry.py:160-193`).
   - Result: the system can open a case family with empty/non-canonical case evidence metrics. This violates the semantic registry as source of truth.

2. **`fulfillment_backlog` exists in semantic/action catalogs but not in `OperationalCaseType`.**
   - `CASE_FAMILY_METRICS` defines `fulfillment_backlog` (`app/brain/semantics/metric_registry.py:177-180`).
   - `action_catalog.py` registers `inspect_pending_orders` for `fulfillment_backlog` (`app/brain/action_catalog.py:189-196`).
   - `OperationalCaseType` currently omits `fulfillment_backlog` (`app/brain/operational_cases.py:28-35`).
   - Result: one semantic case family has actions but cannot be represented as a native Operational Case.

### Required fix

Prioritize a rebased version of `codex/channel-mix-case-gate`:

- Add a canonical `DETECTABLE_CASE_TYPES = frozenset(CASE_FAMILY_METRICS)` or equivalent registry-backed eligibility check.
- Do not promote insight-derived cases whose family has no semantic evidence contract.
- Add `fulfillment_backlog` consistently to the case type catalog if it remains in `CASE_FAMILY_METRICS` and action catalog.
- Add a contract test that every detectable case family is present in `CASE_FAMILY_METRICS`, every `CASE_FAMILY_METRICS` family is representable as an `OperationalCaseType`, and every action catalog `case_families` entry resolves to the same registry.

ARB disposition: **semantic registry is strong, but current case-family drift is a real architecture gap.** Fix before adding more case families or owner-facing case automation.

## 3. Connector platform: adapter / service / storage separation

### Current alignment

The connector platform is now materially better than earlier branches:

- `ConnectorSpec` separates connector metadata, required public config, required secret refs, legacy secret fields, capabilities, executor metadata, health, rate limits, scopes, and lifecycle (`app/brain/connector_registry.py:74-220`).
- Registry-driven adapter kwargs are allowlisted by `ConnectorFactoryParam`; missing required params are reported by key name, not raw values (`app/brain/connector_registry.py:227-309`).
- `compile_business_runtime` is pure and registry-driven; it compiles public params and deterministic `secret://...` refs without embedding raw connector secrets (`app/brain/runtime.py:145-276`, `343-369`).
- Daily report execution uses the connector registry for enabled connector lookup, capability checks, kwargs construction, and factory loading rather than hardcoding every adapter path in the pipeline execution step (`app/brain/pipeline.py`, inspected current base).

### Remaining separation gaps

1. **Preview HTTP endpoints still import and call adapter factories directly.** `server.py` imports adapter builders at module top and calls them in `/brain/reports/daily/*` preview routes (`server.py:18-37`, `551-704`). This is acceptable as legacy preview surface, but new connector routes should go through the registry/runtime service boundary.
2. **Secret resolution is transitional.** The compiled runtime emits secret refs, but `_missing_required_secret_refs` still treats legacy inline params as proof the execution path has a resolvable secret (`app/brain/runtime.py:343-353`). Do not expose this as tenant self-service connector management until a real secret-ref resolver exists.
3. **Connector emitted families are still strings.** The registry validates emitted metrics, but connector-declared `emitted_metric_families` should be contract-tested against semantic registry families/constants before adding more connectors.

ARB disposition: **aligned for current D2C pilot; not yet complete connector platform.** Continue moving preview/runtime execution behind registry services and close the secret-ref execution boundary.

## 4. Workflow automation: idempotency, approval gates, audit

### Current alignment

- `workflow_automation.py` is explicitly a dry-run/projection service: it does not mutate cases, dispatch messages, or call external systems (`app/brain/workflow_automation.py:1-7`).
- Rules are business-scoped and fail closed on business mismatch (`app/brain/workflow_automation.py:156-158`).
- Unknown action keys fail before condition evaluation (`app/brain/workflow_automation.py:166-172`).
- Planned actions include deterministic, redacted idempotency keys (`app/brain/workflow_automation.py:93-111`, `182-218`).
- Duplicate planned actions are suppressed in-memory by idempotency key (`app/brain/workflow_automation.py:252-277`).
- Approval-required and suggestion-only actions remain projections; `side_effects_executed` stays `0` (`app/brain/workflow_automation.py:174-180`, `280-317`).

### Gap fixed by recent branch

`codex/workflow-automation` adds validation that a workflow action with declared `case_families` cannot be planned for the wrong `case_type`. This is the correct direction and should be merged after rebase/CI.

### Remaining automation gaps before any executor

- No durable workflow rule registry/versioning.
- No persisted idempotency reservation/ledger for workflow actions.
- No approval object, approval transition, approver identity, or audit event lifecycle.
- No RBAC check inside workflow planning/execution beyond business scoping.
- No integration with the operator audit store because that store is not yet in current base.

ARB disposition: **dry-run automation is safe; execution must remain blocked** until idempotency, approval, RBAC, and audit are durable and tenant-scoped.

## 5. Trust / Admin / Security

### Current base

Current `feat/orvo-brain-control-plane` still uses a shared bearer token (`ORVO_INTERNAL_OPERATOR_TOKEN`) for internal operator routes (`server.py:145` and related tests). It has good redaction practices in case models, runtime compilation, operator projections, dispatch, and error surfaces, but it does **not** yet have a sellable admin/security boundary:

- No first-class principal model in current base.
- No RBAC/permission enforcement for internal operator mutations.
- No tenant/project entitlement model beyond route `business_id` scoping.
- No append-only operator audit store in current base.
- No audit query/export/retention policy.

### Branch: `codex/trust-admin-security`

What is good:

- Adds `viewer`, `operator`, `admin` role scaffolding and permission checks.
- Adds `SQLiteOperatorAuditStore` with redaction at write time.
- Adds audit events for auth denials, authorization denials, and successful case actions.
- Adds an audit-events read projection.

Why it still needs work:

1. **Stale branch.** It diverges heavily from current action catalog/operator API work and should not be merged directly.
2. **Default role is too privileged.** Missing role defaults to `operator`, which can mutate cases. For Trust/Admin, default should be read-only or explicit claim required.
3. **No business/project-scoped entitlement.** A token holder with an operator role can target any route `business_id` unless a future principal model scopes it.
4. **Post-auth action failures are not comprehensively audited.** Unknown action keys, missing required reason, invalid transitions, and invalid payloads after RBAC should emit denied/failed audit events.
5. **Append-only is conventional, not enforced.** SQLite insert helper is append-only, but schema/triggers/API boundaries do not prevent update/delete by other code paths.

ARB disposition: **needs work / rebase.** Cherry-pick concepts, not the stale branch wholesale.

## 6. Strategic alignment / avoid platform detours

The repo is correctly moving toward an Atlassian-like operations control plane, but the first sellable product remains the D2C ecommerce control plane. Two cautions:

- `codex/edge-developer-platform` introduces good pure policy concepts (business scoping, permissions, idempotency-required decisions, audit event naming). Keep it contract-only until it is wired to concrete D2C operator/runtime routes. Do not market/build it as a generic developer platform.
- `codex/service-management` is useful because it projects existing Operational Cases into JSM-like views without owning lifecycle. Keep it read-only until core workflow/category/SLA contracts are first-class.

## Priority recommendations

1. **Fix case-family registry drift first.** Rebase/cherry-pick `codex/channel-mix-case-gate`; align `OperationalCaseType`, `CASE_FAMILY_METRICS`, and action catalog families.
2. **Merge `codex/workflow-automation` after rebase/CI.** It closes an action-family gating hole while preserving dry-run semantics.
3. **Rebase and merge the owner-facing case brief policy from `codex/work-management`.** It is a good surface guard, but not a substitute for canonical detection gating.
4. **Rework Trust/Admin/Security as a fresh branch on current base.** Use the RBAC/audit scaffold ideas, but add least-privilege defaults, business-scoped principals, failure audit, and append-only guarantees.
5. **Define core work-management registries.** Add Project, IssueType/CaseFamily, WorkflowStatus, and StatusCategory registry contracts before expanding boards/SLA/admin surfaces.
6. **Keep connector runtime work registry-first.** Do not add more adapter-specific server shortcuts; migrate preview paths behind connector registry services over time.
