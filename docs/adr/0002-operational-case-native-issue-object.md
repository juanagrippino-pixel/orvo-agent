# ADR-0002: Operational Case as the native issue object

Status: Proposed
Date: 2026-05-24

## Context

The program plan defines the central product object as an Operational Case: a business-worthy unit of attention such as a risk, deviation, blockage, opportunity, follow-up, approval, or decision point.

The current repo has deterministic reports and insights:

- `app/brain/models.py` defines `Evidence`, `Metric`, `Insight`, `InsightThresholds`, and `DailyReport`.
- `app/brain/insights.py` creates deterministic findings from metrics.
- `app/brain/reporting.py` renders WhatsApp-sized Spanish daily reports.
- `app/brain/dispatch.py` sends a report once using idempotency keys.
- `app/brain/storage.py` persists configs, schedules, and idempotency keys, but not issue/workflow state.

Those objects are necessary but not enough for a control plane. An `Insight` is a finding in a run. A `DailyReport` is a projection. Neither owns lifecycle, assignment, dedupe across days, transitions, comments, audit, or resolution.

## Decision

Make `OperationalCase` the native Orvo issue/workflow object. Reports, WhatsApp alerts, internal queues, timelines, and automations must become projections or actions around Operational Cases.

This is a platform contract, not naming guidance: future `Alert`, `Task`, `Opportunity`, `Incident`, or automation-run concepts may exist only as projections, subtypes, or action records backed by `OperationalCase` identity and evidence. They must not create independent lifecycle stores that compete with cases.

`Insight` remains a deterministic finding object for Hito0 compatibility and for the detection layer. Phase B should introduce case creation from current insights, but the Phase A architecture must reserve the object boundary now.

## Conceptual model

### OperationalCase

An Operational Case represents one active or historical business issue/opportunity that Orvo is tracking.

Required conceptual fields:

- `case_id`: stable identifier generated by Orvo.
- `business_id`: tenant/business owner.
- `case_type`: stable machine key such as `stockout_risk`, `spend_without_orders`, `checkout_failure_proxy`, `data_stale`, `fulfillment_backlog`.
- `title`: current concise label.
- `status`: lifecycle state.
- `severity`: business impact if ignored.
- `priority_score`: ordering score for queues/reports.
- `confidence`: detection confidence derived from evidence quality.
- `entity_scope`: business/channel/SKU/order-flow/campaign scope.
- `opened_at`, `updated_at`, `resolved_at`, `reopened_at` where applicable.
- `owner_ref` or `assignee_ref`: internal operator or business owner routing target.
- `dedupe_key`: stable key for merge/reopen behavior.
- `source_detection_id` or equivalent lineage back to deterministic detection.

### CaseEvidence

Evidence attached to cases must snapshot the facts used to open/update the case.

Required conceptual fields:

- `case_id`
- `evidence_id`
- `source`: connector/source such as `tiendanube`, `meta_ads`, `google_sheets`, `csv`, or `orvo_runtime`.
- `label`: human-readable citation label, compatible with current `Evidence.label`.
- `metric_key`: metric registry key when evidence comes from a metric.
- `metric_value_snapshot`: value at decision time.
- `unit`
- `window_start`, `window_end` where relevant.
- `artifact_ref`: pointer to run artifact / connector execution / raw-normalized snapshot, not raw secret-bearing payload.

### CaseTransition

Transitions are append-only audit events for lifecycle changes.

Required conceptual fields:

- `transition_id`
- `case_id`
- `from_status`
- `to_status`
- `actor_type`: `system`, `operator`, `business_owner`, or `automation`.
- `actor_ref`
- `reason`
- `created_at`
- `run_id` when system-generated.

### CaseComment

Comments are append-only human/system annotations.

Required conceptual fields:

- `comment_id`
- `case_id`
- `author_type`
- `author_ref`
- `body`
- `created_at`

### CaseAction

Actions are requested/performed operations tied to a case.

Required conceptual fields:

- `action_id`
- `case_id`
- `action_type`: e.g. `send_message`, `request_approval`, `create_task`, `mark_follow_up`, `external_action`.
- `status`: queued/approved/running/succeeded/failed/cancelled.
- `requested_by`
- `requested_at`
- `completed_at`
- `result_ref`

## Lifecycle

Initial lifecycle states:

```text
open -> acknowledged -> in_progress -> resolved
open -> resolved
resolved -> reopened
open|acknowledged|in_progress -> dismissed
```

Rules:

- `open` means Orvo has evidence that the business should pay attention.
- `acknowledged` means a human saw it but work has not started.
- `in_progress` means an action or investigation is underway.
- `resolved` means the issue/opportunity is no longer active or the decision was completed.
- `reopened` must preserve the previous resolution history; it is not a brand new case unless the dedupe key materially changes.
- `dismissed` is an explicit human or policy decision that the case is not worth acting on.

## Dedupe and identity

Case identity must prevent daily-report spam.

Recommended initial `dedupe_key` shape:

```text
<business_id>/<case_type>/<entity_scope_kind>/<entity_scope_id>/<primary_metric_key>/<window_grain>
```

Examples:

- `artemea/stockout_risk/sku/SKU-123/stock_units/daily`
- `artemea/spend_without_orders/channel/meta_ads/ad_spend_today/daily`
- `artemea/data_stale/connector/tiendanube/freshness/daily`

Opening logic:

1. If no open or recently resolved case has the same `dedupe_key`, create a new case.
2. If an open case exists, append evidence and update priority/status metadata instead of creating a duplicate.
3. If a resolved case reappears inside a defined recurrence window, reopen it and preserve the old timeline.
4. If the entity scope changes materially, create a different case rather than hiding distinct issues behind one key.

## Relationship to current objects

Current object compatibility:

- `Metric` remains the normalized observation object.
- `Insight` remains the deterministic finding emitted by `app/brain/insights.py`.
- `DailyReport` remains the Hito0 report payload used by existing tests and endpoints.
- `OperationalCase` becomes the durable workflow object generated from deterministic findings and runtime/data-health events.

Projection rule:

```text
Metrics -> deterministic detections / insights -> Operational Cases -> reports, queues, timelines, WhatsApp summaries, automations
```

Reports may temporarily continue to render directly from `DailyReport.insights` during migration. Once case projections exist, owner-facing reports should be sourced from new/open/resolved/high-priority cases plus compact metric summary.

## Storage contract

Phase B storage should be additive to `app/brain/storage.py` and must not break existing tables:

Existing tables:

- `business_configs`
- `report_schedules`
- `idempotency_keys`

Additive tables should be created with `CREATE TABLE IF NOT EXISTS`, following current `init_schema(conn)` style:

- `operational_cases`
- `case_evidence`
- `case_transitions`
- `case_comments`
- `case_actions`

If run ledger tables arrive first, `case_evidence.artifact_ref` should point to ledger artifacts rather than duplicate raw payloads.

## Consequences

### Positive

- Orvo can track unresolved work across days instead of re-announcing the same insight.
- WhatsApp becomes a surface around workflow state, not the product core.
- Internal operator APIs can show queues, timelines, and action history.
- Automation safety improves because actions are attached to auditable cases.

### Negative / costs

- Current report tests will not prove workflow correctness; new contract/invariant tests are required.
- Dedupe and reopen rules introduce product decisions that must be tuned carefully.
- Case priority must be deterministic and explainable; a generic LLM ranking step would break trust.

## Invariants for future workers

1. A case must never exist without evidence.
2. A case transition must be append-only; do not overwrite lifecycle history.
3. A resolved case must remain queryable for audit and recurrence analysis.
4. Case creation must be deterministic from normalized metrics/runtime events and policy thresholds.
5. LLM output may explain a case but must not create, resolve, prioritize, or mutate a case without deterministic/tool-backed evidence and an auditable actor.
6. Case dedupe must be stable across preview, forced run, and scheduled run for the same business/date/input.
7. Owner-facing WhatsApp text must be a projection of case/report data, not a hidden source of state.
8. Existing `DailyReport` behavior must keep passing until the migration explicitly flips a tested projection path.
9. New workflow-like concepts (`Alert`, `Task`, `Opportunity`, `Incident`, automation run) must be projections, subtypes, or actions around `OperationalCase`, not competing issue objects.
